#### 聚集索引 VS 非聚集索引
MySQL 中，B+ 树索引按照存储方式的不同分为聚集索引和非聚集索引
**①聚集索引（聚簇索引）**：以 InnoDB 作为存储引擎的表，表中的数据都会有一个**主键**，即使你不创建主键，系统也会帮你创建一个隐式的主键。
- 这是因为 InnoDB 是把数据存放在 B+ 树中的，而 <u>B+ 树的键值就是主键</u>，在 B+ 树的叶子节点中，**存储了表中所有的数据**。
- 这种<u>以主键作为 B+ 树索引的键值而构建的 B+ 树索引</u>，我们称之为聚集索引。
**②非聚集索引（非聚簇索引）**：<u>以主键以外的列值作为键值构建的B+树索引</u>，我们称之为非聚集索引。
- 非聚集索引与聚集索引的区别在于**非聚集索引的叶子节点不存储表中的数据，而是存储该列对应的主键**，想要查找数据我们还<u>需要根据主键再去聚集索引中进行查找</u>，这个再根据聚集索引查找数据的过程，我们称为**回表**。
- 明白了聚集索引和非聚集索引的定义，我们应该明白这样一句话：数据即索引，索引即数据。
#### 逆范式
一般说来，<u>数据库只需满足第三范式（3NF）就行了</u>。
没有冗余的数据库设计可以做到。但是，没有冗余的数据库未必是最好的数据库，有时为了提高运行效率，就必须<u>降低范式标准，适当保留冗余数据</u>。具体做法是：在概念数据模型设计时遵守第三范式，降低范式标准的工作放到物理数据模型设计时考虑。降低范式就是**增加字段，允许冗余，达到以空间换时间**的目的。
如“金额”=“单价”×“数量”，但是增加“金额”这个冗余字段，<u>可以提高查询统计的速度</u>
> [!NOTE]
> **范式化（时间换空间）**
> 优点：
> 范式化的表减少了数据冗余，数据表更新操作快、占用存储空间少。
> 缺点：
> 查询时需要对多个表进行关联，查询性能降低。
> 更难进行索引优化
> **反范式化（空间换时间）**
> 反范式的过程就是通过冗余数据来提高查询性能，但冗余数据会牺牲数据一致性
> 优点：
> 可以减少表关联
> 可以更好进行索引优化
> 缺点：
> 存在大量冗余数据
> 数据维护成本更高（删除异常，插入异常，更新异常）

#### 不同数据库
**集中式数据库**
- 优点
	- 结构简单，开发/运维容易
	- 性能强且稳定：不需要网络协调；事务、JOIN、聚合都在本机完成，延迟低
	- 强一致性更容易保证
- 缺点
	- 扩展性差：主要靠纵向扩展（加 CPU/内存）单机资源有限
	- 单点故障会带来很大损失，解决方式：热备份或灾难备份
- <u>适用</u>：并发不高或可控、对数据库性能要求高而对扩展性可靠性要求相对低，如学校系统、企业内部系统、中小网站
**分布式数据库**
- 优点
	- 水平扩展能力强
	- 高可用与容灾能力强：多副本机制
	- 适合海量数据和高并发，减少单一节点负担
- 缺点
	- 系统复杂
	- 事务与一致性成本高
	- 跨分片查询代价大：JOIN/聚合/排序可能需要在多节点间交换数据
	- 运维成本高
- <u>适用</u>：要求数据可靠性、并发高写入量大、数据规模非常大，如大型电商、支付系统、日志/行为分析平台
#### 堆文件
堆文件是一种<u>表数据（records/tuples）的物理存储组织方式</u>，**记录之间不按主键/时间/任何字段排序**，像一堆东西无序存放
- 表文件=多个数据页
- 页内结构一般：<u>页头+slot目录+记录区</u>
	- slot 记录“第 i 条记录在页内的偏移”
	- 删除/更新时可以移动记录，但 slot 不变（或可复用）
**插入**
- 找一个有空闲空间的页，直接写入记录，更新页的 slot 信息
- DBMS 通常维护<u>空闲空间目录</u>，以快速找到有空位的页
**查找**
- 没有索引则全表扫描
- 额外建立索引（B+树/Hash）来加速查询
	- 叶结点通常存：<u>(key → RID)</u>，RID = (page_id, slot_id)
**删除**
- 通常<u>将 slot 标记为 deleted</u>
堆文件的随机性使其插入删除更新查询效率都很均衡
#### 数据库各种索引
##### 位图索引
对某个列 `A` 的每一个不同取值 `v`，维护一个**位向量**，<u>维度为行数</u>
![[Pasted image 20260129152038.png]]
用途：
- **低基数列**：字段取值种类少
- 多条件组合过滤特别多（可以做位运算）
缺点：
- **更新代价大**：插入/删除会影响位图
- 高基数列（如手机号、身份证）位图会非常浪费空间
##### 位图联结索引
不只索引一张表的列，而是**把“联结结果”预先编码成位图索引**。把两张表的 join 关系提前算好，建立位图索引，从而让查询时不需要再 join（或减少 join 成本），如对人名建立索引，对象是另一张表（不用再通过uid join获得人名）
#### 关系数据库存树
##### 邻接模型
每个节点存一个 `parent_id` 指向父节点
✅ 优点：
- 表结构简单
- 插入/移动节点方便：只改 `parent_id`
- 适合 OLTP
❌ 缺点：
- 查询整棵子树需要递归/多次 join
- 集合查询性能不佳
**自顶向下查询**
- 不断用 `parent_id` 去找子节点，为 `parent_id` 构建索引
- 不支持递归：只能<u>多次self join或多次查询</u>
- 支持递归
- 一个父节点可以有多个子节点。 自顶向下查询是一个**扇面扩散过程**。 随着深度增加，需要处理的子节点数量可能呈指数级增长，**搜索空间大得多**
**自底向上查询**
- 只沿一条路径向上找
- 复杂度≈树高，较快
**集合查询（查询一个子树所有节点 / 统计子树人数）**
- 必须递归生成整个子树
- join 次数多，结果集大
##### 物化路径模型
每个节点保存从根到自身的路径字符串（或数组），该字段标明层次结构，标识父子关系，固定兄弟间顺序
![[Pasted image 20260131122544.png]]
✅ 优点：
- 子树查询/集合查询极快（LIKE 前缀）
- 查询写法简单
❌ 缺点：
- <u>节点移动成本高</u>：移动一个节点意味着它所有后代 path 都要更新
- path 字符串长度随深度增长
- 自底向上（祖先链）要拆分 path，SQL 不够优雅
**自顶向下查询**
- 变成一次 <u>LIKE 前缀匹配</u>
- 没有递归，没有 join
- 若索引可用于前缀匹配（很多 DB 可优化），非常快
**自底向上查询**
查询 “测试组(id=6)” 的祖先，测试组 path = `/1/2/6/`  
祖先 id 就是路径里的<u>节点集合</u> `{1,2,6}`，看 path 是否容易拆分
**集合查询（子树统计）**
``` sql
SELECT COUNT(*)
FROM dept_path
WHERE path LIKE '/1/2/%';
```
##### 嵌套模型
用两个数字 `lft`（左值）和 `rgt`（右值）表示节点在树中的位置：
- 每个节点占据一个区间 `[lft, rgt]`
- <u>父节点的区间包含所有子节点的区间</u>
- 子树查询就变成了**范围查询**（非常快）
```scss
公司(1)
├─ 技术部(2)
│  ├─ 后端组(4)
│  └─ 前端组(5)
└─ 市场部(3)
```
![[Pasted image 20260131124527.png]]
**自顶向下**
```sql
SELECT child.*
FROM dept_nested AS parent
JOIN dept_nested AS child
  ON child.lft BETWEEN parent.lft AND parent.rgt
WHERE parent.id = 2
ORDER BY child.lft;
```
- 本质是 `BETWEEN` 范围查询
- 可以用 B+树索引快速扫描连续区间
- **非常适合“查整棵子树”**
✅ 结论：嵌套模型在自顶向下查询中性能极强（通常优于邻接表）
**自底向上**
```sql
SELECT parent.*
FROM dept_nested AS node
JOIN dept_nested AS parent
  ON parent.lft <= node.lft AND parent.rgt >= node.rgt
WHERE node.id = 4
ORDER BY parent.lft;
```
- 同样是区间包含判断
- 祖先数量通常较少（树高 logN 级别）
- 也很快，不需要递归
✅ 结论：嵌套模型的自底向上查询也很快
**集合查询**
- 多个范围扫描
- 仍然属于索引擅长的模式
- 查询叶子结点：rgt = lft + 1
✅ 结论：O(N) 扫描，但非常简单，容易优化

#### 锁机制
计算机协调多个进程或线程并发访问某一资源的机制
粒度：行锁，页/块锁，表锁，数据库锁
##### 共享锁 S / 读锁
- 允许多个事务同时加 S 锁读同一数据
- **禁止其他事务加 X 锁修改**
`SELECT * FROM account WHERE id=1 LOCK IN SHARE MODE;   -- MySQL`
- 读操作要求“读期间不被改”
- 例如：对账、生成报表时保证读取的数据不会被并发更新破坏一致性
##### 排他锁 X / 写锁
- 事务加 X 锁后，可读写该对象
- 其他事务不能再加 S 或 X 锁
场景：
- 典型的“先查后改”的并发更新
- 例如：扣余额、减库存、抢票
##### 更新锁
用于<u>“先读后写”</u>场景，避免两个事务都先读加 S 锁，然后都尝试升级为 X 锁造成死锁
- 银行转账、库存扣减等“读后更新”业务
##### 意向锁
意向锁是**表级锁**，表示“我打算在表中的某些行上加锁”，用于解决**行锁与表锁的冲突判断效率**
- IS：意向共享锁（准备对某些行加 S）
- IX：意向排他锁（准备对某些行加 X）
- 事务对某行加 X 锁时，DBMS 会先在表上加 IX 锁
##### 乐观锁
不加锁，一般为数据加一个version字段，每次修改时+1
![[Pasted image 20260201175301.png]]
适用：
- 读多写少 
- 冲突概率低（例如用户修改个人信息）
##### 悲观锁
悲观的认为数据一直不安全，每次数据操作时一定会有其他程序也来修改
- 先锁再操作
- 冲突概率高时更适合（例如抢购）
- 数据库的行锁/表锁通常属于悲观并发控制，如共享锁、排它锁等
- 显示使用：`SELECT ... FOR UPDATE`
