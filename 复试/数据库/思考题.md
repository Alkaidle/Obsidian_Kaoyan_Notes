#### 4. SQL 优化
![[Pasted image 20260117175317.png]]
- `exists = distinct + inner join` 
- JOIN 放大结果集 → 再去重
	- 代价: 结果集放大; distinct 代价: 临时表, 排序 or 哈希
- exists: 对 `film` 逐行扫描, 每行触发一个 **相关子查询
	- 子查询：
	    - 用 `film_actor.film_id` 索引
	    - 找到第一条就返回 TRUE,  立刻停止扫描（短路）
	- 是存在性判断
	- 在一对多关系下，EXISTS 通常能减少 IO 和临时表使用，因此性能更优
- MySQL 对 EXISTS 的处理: 
	- 优化器会：
		- 把 EXISTS 子查询当作 **半连接（Semi-Join）**
		- 不生成完整 JOIN 结果
		- 不返回子表数据
- 子表是 **一对一**, 子表行数极少, 不需要 DISTINCT, 本来就需要 JOIN 出其它字段 时可能用 JOIN

```sql
SELECT o1.custid, o1.ordered, o1.totalitems
FROM orders o1,
     (SELECT custid, MAX(ordered) AS max_ordered
      FROM orders
      GROUP BY custid) x
WHERE o1.custid = x.custid
  AND o1.ordered = x.max_ordered
```
- **从“逐行扫描”到“集合操作”**：原 SQL 是关联子查询，数据库可能需要针对 `o1` 表的每一行都执行一遍子查询去寻找最大值。而改写为内嵌视图后，数据库只需**执行一次**分组聚合运算生成临时结果集 `x`，然后进行高效的表连接
- **利用优化器特性**：根据“越快剔除不需要的数据，效率就越高”的原则，内嵌视图能让优化器更清晰地识别出过滤逻辑，从而选择更优的执行路径（如使用 Hash Join 或 Merge Join）
这种变换体现了关系代数中的**等价变换**思想，通过改变执行方式而不改变结果集，从而在处理大数据量时获得更好的性能
